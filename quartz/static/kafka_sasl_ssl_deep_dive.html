Kafka SASL/SSL authentication configuration — interactive reference

SASL mechanism comparison

PLAIN
Username + password sent in cleartext (must pair with TLS). Simple but credentials are static — rotation requires restart. Only use for dev or internal trusted networks.
SCRAM-256
Challenge-response — password never sent over wire. Credentials stored as salted hashes in ZooKeeper/KRaft. Supports online credential rotation. Recommended for most on-prem clusters.
GSSAPI
Kerberos integration. Uses Kerberos service tickets — no passwords stored in Kafka at all. Best for enterprise Hadoop environments with existing KDC. High ops complexity.
OAUTHBEARER
JWT tokens from an OAuth2 / OIDC provider (Keycloak, Okta, AWS Cognito). Token TTL typically 1h — automatic short-lived credentials. Best for cloud-native and microservice producers.

Listener architecture — separate ports per security level

listeners=PLAINTEXT://0.0.0.0:9092,SASL_SSL://0.0.0.0:9093,INTERNAL://0.0.0.0:9094 # 9092 = internal cluster comms (broker-broker), PLAINTEXT ok inside VPC # 9093 = external clients — SASL + TLS mandatory # 9094 = admin/monitoring tools — separate ACL set listener.security.protocol.map=PLAINTEXT:PLAINTEXT,SASL_SSL:SASL_SSL,INTERNAL:PLAINTEXT inter.broker.listener.name=INTERNAL

Key insight: broker-to-broker traffic can stay on PLAINTEXT inside the VPC. All external producer/consumer traffic hits the SASL_SSL listener only.

server.properties — SASL_SCRAM + TLS broker config

# TLS keystore (broker's own cert) ssl.keystore.location=/etc/kafka/ssl/kafka.server.keystore.jks ssl.keystore.password=${KEYSTORE_PASS} # inject from vault, never hardcode ssl.key.password=${KEY_PASS} ssl.truststore.location=/etc/kafka/ssl/kafka.server.truststore.jks ssl.truststore.password=${TRUSTSTORE_PASS} ssl.client.auth=required # mTLS — clients must present cert too ssl.endpoint.identification.algorithm=https # validate hostname # SASL mechanism sasl.enabled.mechanisms=SCRAM-SHA-256 sasl.mechanism.inter.broker.protocol=PLAIN # broker-broker uses PLAIN on INTERNAL # JAAS config (inline or via -Djava.security.auth.login.config) listener.name.sasl_ssl.scram-sha-256.sasl.jaas.config=\ org.apache.kafka.common.security.scram.ScramLoginModule required; # Authorization authorizer.class.name=kafka.security.authorizer.AclAuthorizer super.users=User:kafka_admin;User:monitoring_svc

Adding SCRAM credentials (stored in ZooKeeper/KRaft)

# Create user credential — salt+hash stored, not plaintext kafka-configs.sh --bootstrap-server broker:9093 \ --command-config admin.properties \ --alter --add-config 'SCRAM-SHA-256=[iterations=8192,password=s3cur3pass]' \ --entity-type users --entity-name spark_etl_svc # Rotate password (zero-downtime — old hash stays valid during TTL) kafka-configs.sh ... --add-config 'SCRAM-SHA-256=[password=new_s3cur3pass]' \ --entity-name spark_etl_svc

Producer / consumer client config (Java properties)

security.protocol=SASL_SSL sasl.mechanism=SCRAM-SHA-256 ssl.truststore.location=/etc/kafka/ssl/client.truststore.jks ssl.truststore.password=${TRUSTSTORE_PASS} # JAAS credentials — pull from vault at startup, not config file sasl.jaas.config=org.apache.kafka.common.security.scram.ScramLoginModule required \ username="spark_etl_svc" \ password="${KAFKA_PASSWORD}"; # For OAuthBearer (cloud-native alternative) # sasl.mechanism=OAUTHBEARER # sasl.oauthbearer.token.endpoint.url=https://keycloak/realms/data/token # sasl.login.callback.handler.class=org.apache.kafka.common.security.oauthbearer...

Spark Structured Streaming Kafka source config

spark.readStream .format("kafka") .option("kafka.bootstrap.servers", "broker:9093") .option("kafka.security.protocol", "SASL_SSL") .option("kafka.sasl.mechanism", "SCRAM-SHA-256") .option("kafka.sasl.jaas.config", s"...ScramLoginModule required username='${secret}' password='${pass}';") .option("subscribe", "orders.raw") .load()

Topic ACL patterns with kafka-acls.sh

# Producer: write to specific topic only kafka-acls.sh --bootstrap-server broker:9093 \ --command-config admin.properties --add \ --allow-principal User:spark_etl_svc \ --operation Write --operation Describe \ --topic orders.raw # Consumer group: read + consumer group offset commits kafka-acls.sh ... --add \ --allow-principal User:ml_pipeline_svc \ --operation Read --operation Describe \ --topic orders.enriched kafka-acls.sh ... --add \ --allow-principal User:ml_pipeline_svc \ --operation Read \ --group ml-consumer-group # Deny a specific IP range (geo-restriction) kafka-acls.sh ... --add --deny-principal User:* \ --deny-host 10.99.0.0/16 \ --operation All --topic pii.events

Ranger for Kafka (replaces native ACLs at scale)

At 50+ topics and 20+ teams, native kafka-acls.sh becomes unmaintainable. Apache Ranger's Kafka plugin gives you: centralized policy UI, audit logs per-operation, group-based policies (not per-user), and deny overrides. Set authorizer.class.name=org.apache.ranger.authorization.kafka.authorizer.RangerKafkaAuthorizer in server.properties — it replaces AclAuthorizer completely.
