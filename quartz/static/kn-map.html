一、数据治理

01 数据治理愿景与痛点 ▾
🎯 愿景
打造企业可信的数据生态，赋能企业创新与可持续发展
⚠️ 三大痛点
1. 数据资产质量：组织庞大，资产复用差，质量较低 2. 标准统一与安全：缺少统一数据标准与计算口径，防止数据泄密 3. 可信任数据：持续评估改进质量；指标保持技术与业务一致理解
02 数据治理思维 ▾
🏗️ 四层架构
数据战略 → 组织架构 → 制度建设 → 平台工具
📋 数据管理八大领域
数据标准管理 数据质量管理 元数据管理 数据架构&模型 主数据管理 数据服务管理 数据安全&隐私 数据生命周期
03 数据治理解决方案 ▾
🛠️ 平台工具层
数据一体化开发工具 数据智能管控工具 数据建模管理工具 大数据智能分析平台
🔧 机制建设
管理办法 · 考核机制 · 运营规范/监督 · 成熟度评估
04 大数据基础平台治理 ▾
❌ 三大问题
1. Spark/Hive/Hadoop 运行参数是否合理影响效率和资源消耗 2. 集群成千上万任务，手工调整参数工作量巨大 3. 异常任务治理：失败、长尾、数据倾斜、资源浪费
💡 技术实现思路
• HBO（History Based Optimization）历史统计优化 • 自动化调优，运行时注入方式，侵入性小 • 灰度验证，多重保护机制 • 成本节省：大数据成本降低 10%~30%
05 大数据基础平台数据预警治理 ▾
⚠️ 痛点
预警冗余 · 无效预警 · 准确性差 · 预警不合理 · 离职交接预警失效
✅ 解决措施
• OA 加强数据交接离职流程 • 预警人员 Backup 机制 • 加强业务含义规则，预警告警等级化落到责任人 • 定期梳理无人处理任务 & 清理频繁重复预警 • 优化集群本身导致的任务故障，加入自动化 hook
06 数仓治理 ▾
📐 数据分层架构
STG 数据贴源层 → ODS 数据操作层 → DW 数据公共层 → DWS 数据汇总层 → DM 数据集市层

二、AB 实验

01 AB实验的发展 ▾
📈 演进历程
单层实验 → 层域多层并行 → 社交/双边复杂场景 → 万级并行实验
🚀 AI 智能化趋势
大模型与机器学习进一步赋能实验自助化、智能化进化
02 实验治理总览 ▾
🏆 实验收益（Leader视角）
Holdout 机制：评估多实验叠加收益；按组织架构设计层级；评估线上实验叠加效果
🔬 实验质量（数科视角）
AA回溯 CUPED 重随机 多重比较修正
💰 成本治理
精准曝光 · 参数增量下发 · bitmap压缩 · Latin分流复用hash · 曝光上报限流
🛡️ 负向治理（Metric Defender）
winner's curse p-hacking SRM pre-experiment bias
03 Holdout 机制 ▾
📌 核心价值
• 目前最接近真实收益的多实验效果叠加评估机制 • 可按组织架构设计不同层级结构，精准评估每个部门收益 • 既可评估已发布实验，也可评估线上正在跑的实验的叠加效果
🏗️ 层级结构示例
Org Holdout → TeamA Holdout → TeamB Holdout layer_A_1 / layer_A_2 / layer_B_1 / layer_B_2 / layer_B_3
04 实验质量 ▾
📊 质量指标维度
维度方法
实验功效AA回溯、CUPED、重随机
正向率结合实验资产，提高假设正向率
FDR控制多重比较修正、Sequential test
SRM 检测随机性检验，确保分流质量
05 成本治理 ▾
📉 四大成本方向
① 曝光上报侧限流：每分流单元每实验每天只需上报一次 ② 分流侧：Latin分流，多实验层复用一次hash；bitmap压缩 ③ 数据链路：Spark参数优化；分桶+bootstrap计算；预计算跨实验打包 ④ 实验限制：控制滥用 · 历史数据过滤假设 · 限制指标数量
06 负向治理 ▾
🚨 全局监控体系
• 建立全局监控指标组，提供主动探索工具 • 自动报警机制，按大盘影响排序召回 • 根据用户反馈调整报警规则，告警通知合并
🎯 精准率优化
结合实验资产沉淀，挖掘历史报告，总结归因链路，降低实验创建门槛
07 实验智能化 ▾
🤖 AI 赋能方向
大模型结合 MAB（多臂老虎机） 实验仿真 参数自动调优 对话式实验报告解析
💡 在线实验调参
根据用户 context 决定最佳投放策略；离线 Spark 链路优化

四、DataOps

01 数据开发的困境 ▾
⚠️ 三大核心问题
① 数据所有权问题：业务负责人拥有数据所有权，但受限于经营和组织压力，投入精力有限 ② 数据价值问题：最终消费者离开发者有距离，开发团队难以与业务价值直接关联 ③ 数据专业性问题：成本、效率度量困难；安全问题日益突出
02 DataOps的目标 ▾
🎯 三大目标
① 明确核心数据资产及相关任务，映射线上功能，体系化核心决策指标 ② 让核心数据资产具有持续主动迭代能力 ③ 核心数据资产具备完整的自动化建设能力
03 DataOps的组织 ▾
🏢 数据所有者
认可数据价值，明确谁为业务数据负责，做好业务管理与数据体系结合
🔧 数据生产者
做好元数据的提供，关心数据全链路，理解数据变更对下游的影响
💻 数据开发者
持续提升研发效率，以数据消费为核心业务价值，主动解决业务问题
👤 数据消费者
拥抱数据，主动使用数据，以用数思路刻画业务流程
04 DataOps的实现 ▾
🔄 数据需求循环
问题提出 → 需求提出/分配 → 需求评审 → 模型评审 → 数据开发 → 数据测试 → 数据上线 → 数据验收 → 数据消费 → 问题解决
🛠️ DataOps 工具体系
需求管理平台 模型平台 数据开发平台 资产平台 全流程需求绑定 模型检测 代码检测 自动测试 发布流水线
05 DataOps的难点 ▾
⚡ 四大挑战
① 组织广泛认同：避免纯研发视角，以业务价值打动数据所有者 ② 工具集成：避免短视，关注完整工作流，评估对开发效率的负面影响 ③ 改造现有流程：不同团队流程尽可能敏捷统一，有明确终极目标 ④ 打通消费场景连接：避免自嗨，关注数据消费者实际感受
06 DataOps的未来 ▾
🚀 开发效率方向
• 围绕可持续迭代的数据资产，全面提升自动化水平 • 基于线上功能上线自动实现从数据集成到指标提供
🤖 消费效率方向（AI驱动）
• 大模型数据问答是发展趋势 • 进一步丰富元数据，持续追踪数据产生背景 • 数仓模型层适配大模型自动生成 SQL，明细层为主、汇总层为辅
