01 — 核心抉择

为什么"竖着存"

同一张表，可以按行挨个写到磁盘，也可以把每一列的值聚在一起写。这个布局选择，决定了一次分析查询要从磁盘搬运多少字节。下面这张三行表是同一份数据，切换布局看磁盘上字节的真实排列，再点查询看引擎读了哪些块。

点击运行，看引擎为了求 avg(age) 究竟从磁盘读了哪些块。

分析型查询(OLAP)往往只关心几列、却要扫上亿行。列式布局让"只读 age 列"成为可能 —— 同列数据类型一致、取值相近，还能被极致压缩。代价是:写入单行、或要取"整行所有列"时，列式要东拼西凑，不如行式直接。这正是 OLTP 用行存、OLAP 用列存的根因。

02 — 物理结构

一个 .parquet 文件里有什么

这是 Apache Parquet 官方的文件结构图,这里把它重绘成可交互版本。左侧是磁盘上的物理布局:文件被横向切成 Row Group(行组),组内每一列连续存为 Column Chunk,再细分成 Page(页)—— 页才是编码与压缩的最小单元。右侧 Footer 是元数据中枢。精髓在中间的箭头:Footer 用字节偏移(offset)给左侧每一块做索引。悬停或点击右侧的元数据块试试 ↓

悬停或点击右侧 Footer 中的任一元数据块,看它在左侧物理布局中实际指向的位置。这正是 Footer 作为"索引"的核心作用。

反直觉之处:统计信息要等数据全写完才能算出,所以元数据写在文件末尾。引擎读文件先 seek 到结尾读 Footer,据此精准跳转,无需扫描全文。Footer 末尾还记着自身长度(上图 Footer length),因此读取从文件最后 8 字节开始倒着来 —— 这也是 Parquet 适合"一次写、多次读"的不可变存储的原因。

04 — 数据模型

最烧脑的一块:嵌套怎么拍平成列

列式存储的难题是:嵌套和数组怎么办? 一个人有 0 个或多个电话,光存值序列无法还原"哪些电话属于哪个人、谁的电话是空的"。Parquet 借用 Google Dremel 的方案,给每个值附带两个小整数 —— Repetition Level(R) 和 Definition Level(D) —— 就能无损拍平再还原。

Schema

message Contact {
  required string name;
  repeated string phone;  // 0..n 个电话
}

三条原始记录 —— 点任意一条,看它的 phone 列被拍平后是哪几行(高亮),以及 R/D 的含义:

① { name: "Alice", phone: ["111", "222"] }
② { name: "Bob",   phone: [ ] }  ← 空数组
③ { name: "Carol", phone: ["333"] }

phone 列在磁盘上(拍平后)

来自记录Value 值RD

R(Repetition Level) = 这个值在第几层嵌套上"重复"了上一个值。R=0 表示开启一条新记录;R=1 表示是同一条记录里数组的后续元素。D(Definition Level) = 路径上有多少个可选/重复字段"真的存在"。当 D 没达到最大值,说明这里是 NULL 或空 —— 此时不存真实值,只靠 D 还原。靠这两列整数,引擎能在完全不存 NULL 占位、不存结构标记的前提下,精确重建任意深度的嵌套。

05 — 查询加速

元数据让引擎"跳过"大半数据

前面 Footer 里每个 Column Chunk 都带了 min/max 统计。这意味着:带 WHERE 条件的查询不用扫数据,先看统计就能判断"这一块根本不可能命中,直接跳过"。这就是 谓词下推(Predicate Pushdown) + 行组裁剪。改下面的条件值,看哪些行组被跳过。

SELECT * FROM events WHERE age >

再叠加列裁剪(Column Pruning):如果查询只 SELECT age,引擎连其他列的 Column Chunk 都不会去读。两招合起来 —— 跳过无关行组、跳过无关列 —— 一次扫描可能只触碰原始文件的百分之几。这就是 Parquet 在分析场景下快得反直觉的根本原因。
